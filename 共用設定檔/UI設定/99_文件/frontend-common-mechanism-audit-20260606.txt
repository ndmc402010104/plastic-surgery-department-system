用途：前端共用機制現況盤點 audit；只記錄靜態搜尋與讀碼結果，不修改正式功能、不建立正式 App Shell 架構。

一、總覽

1. 已存在的共用機制：共用設定檔/AppShell.html、EnvironmentFooter.js、Config.js、Route.js、UI設定系列元件、version.json 與 Update-VersionManifest.ps1。
2. Header 已有 AppShell.html，但 GitHub Pages 版多頁仍自行注入 ghHeader；subtitle、固定/相對定位、body padding 與寬度補償不一致。
3. Footer 已有 EnvironmentFooter.js 與 Apps Script 端 getVersionBadgeHtml()，但同時存在 favicon.js 自動載入、各頁 fallback 載入與 Route.js 內嵌 footer HTML，屬於半統一。
4. Version 已有 version.json manifest 與 push.ps1 / Update-VersionManifest.ps1，但 EnvironmentFooter.js 仍保有硬編碼版本，Apps Script 測試版不讀 version.json。
5. Runtime 判斷已集中在 EnvironmentFooter.js/Route.js/Config.js 一部分，但多頁仍重複判斷 github.io、dev-skhps.jonaminz.com、skhps.jonaminz.com、script.google.com。
6. URL/Route 已有 GITHUB_PAGE_ROUTES、APP_SCRIPT_PAGE_ROUTES 與 footer navigation helper，但各頁仍有 GITHUB_PAGE_MAP、getGithubPageUrl、goHome/goPage 的重複實作。
7. 後端呼叫 adapter 已在 HIS、QR、院內登入、敷料頁各自存在，但不是全站共用；google.script.run、JSONP、fetch no-cors 混用。
8. Loading/Error/Toast 已有 UI 元件雛形，但正式頁多用自製 notice、wait indicator、status 或 alert，錯誤診斷資訊不一致。
9. Table/RWD 已有 UiTable.html 與測試頁 pattern，但敷料、後台、簽到頁仍各自用 table/min-width/overflow-x；敷料手機版欄位仍有跑版風險。
10. 最適合優先整理的是只讀/外殼類：footer script 載入規則、runtime detect helper、route map helper、error diagnostics schema、loading indicator 樣式；暫時不該碰相機掃描、敷料送出、QR 簽到驗證、HIS 真實對接與 Apps Script action。

四、最值得優先整理的項目

1. Footer 載入規則與重複注入防護
   為什麼值得先做：影響多頁但可只改載入與 data attribute，不碰業務流程。
   可能影響頁：FrontIndex、SignQRGenerator、HospitalSignIn、DressingUse、DressingFront、DressingDict、skh-ui-test-page、favicon.js。
   最小修改：先建立「每頁只能有一個 data-skh-environment-footer script、一個 data-app-footer」的檢查，不改版面。
   Regression checklist：footer 不重複、手機高度一致、footer 不遮住內容、版本按鈕三欄仍可點、GitHub Pages 與 Apps Script 都顯示。

2. Runtime detect helper 文件化與測試頁 mock
   為什麼值得先做：目前 env 判斷分散，是 URL 跳錯環境與 API 打錯部署的主因。
   可能影響頁：所有 GitHub Pages 前端與 EnvironmentFooter.js。
   最小修改：先不替換正式頁，新增 App Shell Route Test Page mock 各 hostname/search/path 判斷結果。
   Regression checklist：dev-skhps 不被視為 prod、github.io 行為明確、script.google.com /dev 與 /exec 文案正確、appEnv 參數保留。

3. Route map 對照與 page key 正規化
   為什麼值得先做：多頁 GITHUB_PAGE_MAP 重複，且 page key 大小寫不一致。
   可能影響頁：FrontIndex、DressingUse、DressingFront、SignQRGenerator、HospitalSignIn、HisConnect、AdminIndex。
   最小修改：先產出 route map audit 或 helper read-only compare，不直接改 onclick。
   Regression checklist：首頁、QR、院內登入、敷料查詢、敷料領用、UI測試、HIS、後台在 dev/prod/gas 三環境都不互跳錯。

4. Error diagnostics 顯示規格
   為什麼值得先做：HIS 與 API 載入失敗目前難判斷哪一層壞；規格可先做低風險顯示。
   可能影響頁：HisConnectPage、SignQRGenerator、HospitalSignIn、DressingUse、DressingFront。
   最小修改：先在 HIS 測試頁 mock 顯示 action/stage/env/time/version/apiUrl，不碰後端。
   Regression checklist：錯誤包含 action、階段、env、時間、版本；不暴露敏感 token；手機版可讀。

5. Loading indicator 名稱與阻擋原則
   為什麼值得先做：AppWait、waitIndicator、setBusy、loading class 多套，且敷料掃碼被 dataReady 阻擋。
   可能影響頁：DressingUse、DressingFront、SignQRGenerator、HospitalSignIn、AdminMeeting。
   最小修改：先定義「全頁阻擋、按鈕 loading、欄位 loading、資料預載」四種狀態，不先替換核心流程。
   Regression checklist：敷料領用基本資料可先填、掃碼不被不必要初始化阻擋、送出時仍防重複提交、按鈕寬度不跳動。

五、暫時不建議整理的項目

1. 相機掃描流程：DressingUse、DressingFront、DressingDict 使用 Html5Qrcode，含 stop/clear、active row、overlay、手機權限；App Shell Router 下 cleanup 風險高，應逐頁導入。
2. 敷料領用核心送出流程：submitUseForm 會驗證、確認、扣庫、重複領用警示、送出後 refresh inventory；不應在共用 adapter 尚未穩定前改。

5. 後端 Apps Script action：Route.js doGet/doPost、DressingInventory、DressingBarcodeFetch、HisBridgeService 涉及資料寫入與部署 endpoint，本輪不改。
6. 後台 AdminMeeting：仍是 Apps Script only 且大量 google.script.run、alert/confirm；不適合第一波 GitHub Pages adapter。

七、Regression checklist 建議

1. Footer 比例一致：桌面與手機皆只有一個 footer；三欄按鈕等寬；不遮住主要按鈕、表格底部與 scanner overlay。
2. Version 顯示一致：dev-skhps 顯示測試版、本頁版本與正式最新版；skhps 顯示正式版、本頁版本與測試最新版；gasDev 顯示 Apps Script 測試版。
3. Runtime 顯示一致：script.google.com/dev、script.google.com/exec、dev-skhps.jonaminz.com、skhps.jonaminz.com、github.io 都有明確 env。
4. URL 不跳錯環境：從 dev-skhps 切頁不跳 skhps；從 skhps 不跳 dev；Apps Script /dev 保留 appEnv=dev/page；/exec 顯示警示或導向規則符合預期。
5. callBackend 不破壞 Apps Script / GitHub Pages 分流：Apps Script 版使用 google.script.run；GitHub Pages 使用 API URL/JSONP/fetch；回傳 ok/message/data 格式一致。
6. Loading 不阻擋掃碼：資料預載可以顯示進度，但不應不必要阻擋基本輸入；若掃碼必須等資料，要明確說明且可重新整理。
7. Error 能顯示 action / 階段 / env / 時間：至少 HIS、敷料 API、QR 會議清單、院內登入清單要能看出失敗層。
8. 手機版 header/footer 不遮住內容：720px/640px/480px 寬度檢查；header 高度、body padding、footer 高度、fixed element z-index 都要測。
9. 事件不重複綁定：重跑 App Shell render 或 hashchange 後，不重複 click/input/interval/scanner listener。
10. Scanner cleanup：離開頁面、關閉 overlay、重新開啟 scanner 後 Html5Qrcode stop/clear 正常，無殘留相機。
11. Table/RWD：敷料名稱、院內碼、庫存狀態、自費價格在手機版不可互相覆蓋；水平捲動提示可見；重要欄位可讀。
12. Toast/Modal：手機版不被 footer/header 擋住；alert/confirm 替換前要確認流程不改變。
13. 不影響現有頁面功能：QR 產生、QR 簽到、院內登入、敷料領用、敷料查詢/建檔、HIS 測試、後台簽到管理皆須保留既有入口與 action。
14. Version manifest：version.json 讀取失敗時 footer fallback 文案清楚，不造成頁面 JS error。
15. GitHub Pages base path：github.io repo prefix 與 custom domain 根目錄都能正確組路徑。
